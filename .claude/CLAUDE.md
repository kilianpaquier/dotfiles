# Global instructions

## AI behavior

### Response

- Always respond in English.
- Be concise. Short responses unless the task genuinely requires length.
- Don't explain what code does - well-named identifiers do that. Only explain the non-obvious what or why.
- Don't suggest follow-up tasks unless asked.
- No unsolicited affirmations or closing pleasantries.

### Scope

- Never create documentation files (README, CHANGELOG, etc.) unless explicitly asked.
- No git actions (commit, push, PR) unless explicitly asked.
- No unsolicited features, refactoring, or abstractions. Do exactly what was asked.
- Prefer editing existing files over creating new ones.

### Process

- Follow instructions (projects, files specifics, global) to the letter.
- Always verify assumptions before stating them: use Context7 first, then MCP tools, then web search.
- Never assert a capability or compatibility claim about any external tool, framework, library, or service without prior verification.
- Ask before acting when the request or implementation plan is ambiguous, introduces refactoring or anything structural.

## Code review

- Review only when the user explicitly asks — do not offer, suggest, or initiate reviews spontaneously.
- When using sub-agents to review, provide this whole section as additional input context.
- Always review each modified file in full (not just the changed hunks) to catch inconsistencies, duplication, and coherence issues within the file.
- Focus on:
  - Architecture and design: structure, abstraction boundaries, separation of concerns.
  - Bugs and functional correctness.
  - Security issues.
  - Adherence to project conventions.
  - Unnecessary complexity or duplication.
  - Documentation: missing or incorrect language-mandated comments (GoDoc, JSDoc, etc.).
- Skip generated, vendored, and third-party code (dirs: `vendor/`, `node_modules/`, `dist/`, `build/`; files with `Code generated by` / `DO NOT EDIT` / `@generated` headers). Strip those hunks from the diff before passing to any sub-agent.
- Flag dead and unreachable code as a finding.
- Apply the same focus to test files; additionally check language-specific test conventions (e.g. go-test.md rules for package naming, helper markers, phase comments).

## Code style

- Always use English for comments, documentation, and commit messages.
- Default to no inline comments. Add one only when the WHAT or WHY is non-obvious (hidden constraint, subtle invariant, bug workaround).
- Follow language documentation standards on every exported identifier: GoDoc for Go, JSDoc with @param/@returns/@throws for TypeScript.
- Order fields/blocks alphabetically by default (struct fields, interface methods, HCL blocks, object keys, etc.) unless a language rule file defines an explicit order (e.g. GitLab CI job keys, Docker Compose service keys).

## Design

- Don't add defensive checks for inputs or states that are structurally impossible. Always handle real error return values.
- Keep things straight and simple.
- No backwards-compatibility shims or stubs for removed code.
- No premature abstractions or interfaces. Use a concrete type; introduce an interface only when multiple distinct implementations exist.

## Testing

- Test behavior, not implementation details.
- Don't add tests for trivial or unreachable code paths.
- Keep tests minimal and focused - one behavior per test.
- Mark test phases with `// Arrange`, `// Act`, `// Assert` comments (ignore the comment when ther're no steps in given test phase).
